# AGENTS.md

本文件总结本项目协作中的开发原则，供后续 AI/开发者统一执行。

## 1. 交付流程

- 默认流程：`改动 -> 验证 -> commit -> push -> 部署`。
- 若用户明确说“先不要上线/不要部署”，才暂停部署。
- 每次需求完成后都要有清晰结果回报（改了什么、是否通过验证、是否已部署）。

## 2. 代码与改动原则

- 以最小可行改动满足需求，避免无关重构。
- 不回滚用户已有改动，不做破坏性 Git 操作（如 `reset --hard`）。
- 有功能回归风险时先做构建或必要验证。
- 发现问题先修复再继续推进，并说明原因。

## 3. 前端体验原则

- 需求以用户指定样式和布局为准，严格按位置/顺序/主题执行。
- 三套主题（含黑夜模式）都要检查，避免文字与背景对比问题。
- 列表和筛选规则要准确（例如 `#` 分类在 `A-Z` 后）。
- 避免卡顿：优先减少不必要渲染、避免重计算、控制大列表展示成本。

## 4. 数据与脚本原则

- 批量数据脚本必须可重复执行（幂等）并带去重逻辑。
- 补专辑优先录音室专辑（Studio Album），尽量避免错绑到同名乐队。
- 封面来源优先使用网易云（126）；能用 126 就用 126。
- 用户要求远程 URL 时，优先写 126 远程 URL；仅在必要时使用本地文件兜底。

## 5. 权限与管理约束

- 管理权限和特殊入口按用户要求执行（如仅特定用户可操作）。
- 涉及用户可见行为（公告、删除、权限）时，优先保证安全边界和可追溯性。

## 6. 沟通原则

- 用简洁中文同步进度，先说行动再说结果。
- 出现阻塞（网络源不可用、数据源缺失、权限限制）要立即说明并给替代方案。
- 对“先回答我”“先不要上线”等指令优先级最高。

## 7. 测试与资源保护原则

- 默认低负载验证：先静态检查、再最小范围测试，避免直接跑全量任务。
- 测试命令必须控制资源占用（低并发、限速、分批），避免 CPU/内存突增。
- 涉及大数据量或长时间任务时，先做小样本验证，通过后再逐步扩大范围。
- 执行可能高负载操作前，需先说明计划；高风险压测类操作必须先得到用户确认。
- 监控到 CPU/内存异常时应立即停止当前任务，并及时向用户同步。
